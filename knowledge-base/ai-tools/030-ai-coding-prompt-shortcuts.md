## AI 編程常用快捷短語

| 短語 | 用途 |
|------|------|
| 我們討論一下，不要改任何代碼 | 防止 AI 在理解階段直接動手改代碼 |
| 你明白我的需求吧？ | 確認 AI 理解需求後再執行 |
| 畫個 ASCII 圖看下效果 | 先視覺化預覽設計，避免盲目實作 |
| 查看下 git 改動，有什麼地方改錯了導致這個 bug | 利用 git diff 追溯錯誤來源 |
| 把上面討論的內容整理成文檔放在 docs/ | 將討論結果文檔化 |
| 把上面的討論規範寫成規則放入 rules/ | 將最佳實踐固化為專案規則 |
| 請改成 shadcn 的組件，不要使用自定義顏色 | 統一使用設計系統組件 |
| 請使用 pnpm 不要使用 npm | 指定特定工具避免環境不一致 |
| 請把這個做成可復用的組件 | 提升代碼復用性 |

## Specification Mode：規劃優先的開發模式

### 傳統 Vibe Coding 的問題

直接口述需求給 AI Agent，它立即開始改代碼，常見後果：

- ❌ 規劃不全，架構歪樓
- ❌ 產生一堆 bug，甚至把項目搞崩
- ❌ 缺乏測試覆蓋
- ❌ 安全隱患未考慮

### Factory Droid 的 Specification Mode 工作流程

**1. 輸入階段**（4-6 句自然語言描述需求）

```
範例：添加用戶登錄系統，支持 OAuth 和郵箱驗證
```

**2. 規劃階段**（純讀模式，不碰代碼）

Droid 自動執行：
- 深度分析整個代碼庫
- 參考 CLAUDE.md 中的項目規範
- 生成完整規格說明文檔

**3. 規格說明包含內容**

| 項目 | 說明 |
|------|------|
| Acceptance Criteria | 驗收標準 |
| Implementation Plan | 完整實現計劃 |
| File-level Breakdown | 文件級修改分解 |
| Testing Strategy | 測試策略 |
| Security & Compliance | 安全與合規檢查 |

**4. 審核與執行**

- 開發者審核規格後決定執行級別：
  - **Low**：一步步確認每處修改
  - **Medium/High**：更自動化執行，但仍可控

### 核心優勢

| 優勢 | 說明 |
|------|------|
| ✅ 架構更合理 | 先想清楚再動手，避免低級錯誤 |
| ✅ 代碼質量提升 | 完整的測試覆蓋和質量檢查 |
| ✅ 企業級安全 | 審計日誌、外部集成安全考量 |
| ✅ 大型項目可控 | 複雜 Agent 項目終於可以穩定推進 |

### 使用方式

- 在 Factory Droid 中按 `Shift+Tab` 啟動 Specification Mode
- 規劃階段絕不碰一行代碼，零風險
- 適合複雜 Agent 項目的工程化開發
